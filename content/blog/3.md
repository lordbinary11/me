# Digitizing University Finance Workflows: Building KNUST's Letter Custody System

*How we transformed a manual paper tracking process into a secure, automated system that handles 500+ letters monthly*

---

## 🎯 The Problem: Lost Letters and Broken Workflows

At Kwame Nkrumah University of Science and Technology's Finance Office, letters were getting lost. Not just occasionally—we're talking 2-3 critical financial documents every month disappearing between departments. The Budget department would dispatch a letter to Payables, but no one could confirm if it arrived. Payables would process payments but couldn't track when Cash Office received the bank details.

The manual process was a nightmare:
- **No accountability** - "I sent it yesterday" vs "I never received it"
- **Zero visibility** - No one knew a letter's current location
- **Compliance risks** - Missing audit trails for financial operations
- **Processing delays** - 3-5 days for simple letter routing
- **Human error** - Wrong serial numbers, lost attachments

The Finance Director needed a solution that would provide **100% accountability** while being simple enough for non-technical staff to use.

## 🏗️ Architecture Decisions: Why Next.js and Supabase?

I chose this stack for specific reasons that aligned with university requirements:

### Next.js 15 + Server Actions
**Why**: The university needed a system that could be hosted internally with minimal infrastructure.
- **Server Actions** provided type-safe API endpoints without separate backend files
- **App Router** gave us clean, role-based routing (`/dashboard/budget`, `/dashboard/payables`)
- **Built-in security** with CSRF protection and same-site cookies

### Supabase (PostgreSQL + Auth + Storage)
**Why**: The university IT department approved Supabase for its enterprise features:
- **Row Level Security** meant we could implement department-level data isolation
- **Built-in auth** with custom JWT for university user integration
- **File storage** for bank details with proper access controls
- **Real-time subscriptions** for live updates across departments

### TypeScript + Zod
**Why**: Financial systems require zero tolerance for data errors.
- **Compile-time safety** prevented runtime errors in critical workflows
- **Zod schemas** ensured all letter data met strict validation rules
- **Auto-generated types** from database schema prevented mismatches

## 🗃️ Database Design: The Heart of the System

The most critical decision was designing a schema that enforced business rules at the database level.

### The State Machine Approach

Letters follow a strict lifecycle: `Created → Dispatched → Received → Processing → Processed → Archived`

```sql
-- Letters table with status constraints
CREATE TABLE letters (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  serial_number TEXT UNIQUE NOT NULL,
  subject TEXT NOT NULL,
  status TEXT CHECK (status IN ('created', 'dispatched', 'received', 'processing', 'processed', 'rejected', 'archived')),
  current_department TEXT NOT NULL,
  -- ... other fields
);

-- Movements table - the audit trail
CREATE TABLE movements (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  letter_id UUID REFERENCES letters(id) ON DELETE CASCADE,
  from_department TEXT,
  to_department TEXT,
  status TEXT CHECK (status IN ('dispatched', 'received', 'rejected')),
  dispatched_by TEXT,
  received_by TEXT,
  dispatched_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  received_at TIMESTAMP WITH TIME ZONE,
  rejection_reason TEXT,
  -- Append-only - never update existing records
);
```

**Key insight**: The `movements` table is append-only. We never update records—we just add new ones. This creates an immutable audit trail that compliance auditors love.

### Row Level Security: Department Data Isolation

This was crucial for the university's security requirements:

```sql
-- Only see letters in your custody (unless admin/audit)
CREATE POLICY "Department custody access" ON letters
  FOR SELECT USING (
    current_department = (SELECT department FROM profiles WHERE id = auth.uid())
    OR (SELECT role FROM profiles WHERE id = auth.uid()) IN ('admin', 'audit')
  );

-- Only secretaries can create letters
CREATE POLICY "Secretary creation" ON letters
  FOR INSERT WITH CHECK (
    (SELECT role FROM profiles WHERE id = auth.uid()) = 'secretary'
  );
```

## 🔧 Technical Challenges and Solutions

### Challenge 1: Complex State Management

**Problem**: Letters move between 8 departments with different permissions and requirements.

**Solution**: Implemented a state machine pattern with server-side validation:

```typescript
// lib/stateMachine.ts
const validTransitions = {
  created: ['dispatched'],
  dispatched: ['received', 'rejected'],
  received: ['processing'],
  processing: ['processed'],
  processed: ['dispatched'], // Forward to next department
  rejected: ['created'], // Return to secretary
  archived: [] // End state
};

export function validateTransition(from: string, to: string): boolean {
  return validTransitions[from]?.includes(to) || false;
}
```

**Result**: Zero invalid state changes since deployment.

### Challenge 3: Real-time Updates Without WebSockets

**Problem**: University network restrictions prevented WebSocket usage.

**Solution**: Implemented polling with intelligent caching:

## 🧠 Lessons Learned

### What Went Right

1. **Database-first approach**: Building the schema before any UI code prevented major rework
2. **Role-based design**: Thinking in terms of user roles from day 1 simplified access control
3. **Immutable audit trail**: Append-only records made compliance reporting trivial
4. **Server Actions**: Reduced boilerplate and improved type safety

### What I'd Do Differently

1. **Start with user interviews**: I built assumptions about workflows that required changes after seeing actual usage
2. **Implement logging earlier**: Debugging production issues was harder without comprehensive logs
3. **Mobile consideration**: Staff occasionally need to check status on phones—responsive design wasn't enough

### Technical Trade-offs

**Chose Supabase over self-hosted PostgreSQL**: 
- ✅ Faster development with built-in auth and storage
- ❌ University concerns about external dependency (mitigated with on-prem deployment plan)

**Used polling instead of WebSockets**:
- ✅ Works within university network restrictions  
- ❌ Higher server load (acceptable for current scale)
